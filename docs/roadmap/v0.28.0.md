# v0.28.0 — Writer Fencing & Concurrency Correctness

> Replaces the wall-clock millisecond writer epoch with a collision-proof
> monotonic counter, makes GC lease/pin enforcement fully transactional, and
> wraps `rebuild_catalog()` in an atomic WriteBatch. All P0/P1 concurrency
> correctness findings from Assessment Report 1 are resolved.

## Changes

### Monotonic Writer Epoch (P0)

`CatalogStore::open()` previously derived `writer_epoch` from
`SystemTime::now().as_millis()`. Two writers opening in the same millisecond
could compute identical epochs and both pass `check_epoch()`.

**Fix**: The epoch is now a persisted monotonic counter stored at
`SYSTEM_WRITER_EPOCH`. Each open() atomically reads, increments, and writes the
next epoch via a CAS loop — the second concurrent opener always gets epoch N+1
while the first holds epoch N. A UUID nonce (`SYSTEM_WRITER_NONCE`) is written
alongside the epoch so two writers that theoretically obtain the same counter
value (impossible with the CAS loop, but belt-and-suspenders) are still
distinguished at commit time via `check_epoch()`.

**Impact**: The `tokio::time::sleep(Duration::from_millis(2))` calls that
existed as ordering workarounds in `concurrent_writer_fencing.rs` and
`v028_atomicity_tests.rs` have been removed. Fencing is now deterministic with
no wall-clock dependency.

### Transactional GC Lease/Pin Enforcement (P1)

`gc_apply()` correctly documented a single serializable transaction but the
pin scan (`read_pinned_snapshots`) and lease scan (`minimum_leased_snapshot`)
were called against the `Db` handle **outside** the transaction, creating a
TOCTOU window: a lease acquired between the scan and the retain-from write
would not be accounted for.

**Fix**: Added `read_pinned_snapshots_in_tx(tx: &DbTransaction)` in `gc.rs`
and `minimum_leased_snapshot_in_tx(tx: &DbTransaction)` in `lease.rs`. Both
use `tx.scan_prefix()` so the scans run within the same `SerializableSnapshot`
transaction as the retain-from write.

### Atomic `rebuild_catalog()` (P1)

`rebuild_catalog()` previously issued sequential `db.put()` calls. A crash
between any two writes would leave a partially-rebuilt catalog with no way to
distinguish a complete catalog from a partial one.

**Fix**: All rows (schema, table, data files, snapshot, counters) are now
staged in a `WriteBatch` and committed atomically via `db.write(batch)`. The
catalog is either fully present or fully absent after any crash point.

## New Tests

| Test | Description |
|------|-------------|
| `no_sleep_same_tick_exactly_one_fenced` | Two writers opened with no sleep — deterministic fencing without wall-clock |
| `gc_lease_acquired_concurrently_is_respected` | Lease acquired before `gc_apply` correctly blocks GC |
| `partial_rebuild_leaves_catalog_absent` | WriteBatch dropped before commit leaves DB empty |
| `rebuild_catalog_is_all_or_none` | Successful `rebuild_catalog()` produces fully-readable catalog |
